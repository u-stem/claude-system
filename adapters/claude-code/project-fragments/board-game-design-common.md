# 板ゲー設計共通指針

ボードゲーム / カードゲーム / TRPG 等の卓上ゲーム設計の共通指針(物理 / デジタル両用)。
ルールブック執筆 / バランステスト / プレイテストログの三本柱で運用する。

## 設計プロセス

- **コアメカニクスを 1 文で書ける**まで研ぎ澄ます(例: 「資源を集めながら他プレイヤーを妨害して勝利点を稼ぐ」)
- プレイ時間の目安(例: 30 分 / 60 分 / 90 分)とプレイヤー人数(例: 2-4 人)を最初に固定
- 勝利条件と終了条件を分けて定義(終了条件 = 勝者を判定するタイミング、勝利条件 = どう勝つか)
- ターン構造(ラウンド・フェイズ・アクション)を ASCII 図 or table で文書化

## コンポーネント設計

- カード / トークン / ボード / ダイス等のコンポーネント数を **最小限**に保つ(セットアップ・片付けの負荷)
- 同じ機能のコンポーネントは色 / アイコン / 形状で区別、テキスト依存を減らす(言語非依存性 / 視認性)
- 紛失しやすい小駒は予備を多めに見積もる
- カードのテキスト量は**短く**、複雑効果はキーワード化(「先制」「カバー」等)

## バランステスト(数値設計)

- コスト ↔ 効果のマトリクスを spreadsheet で管理(`docs/balance-tests/` 配下に配置)
- 各カード / 各ユニットのレート(コストに対する勝率 / 期待値)を計算
- モンテカルロシミュレーション(確率系の検証)/ 小規模 AI 対戦(戦略系の検証)で初期バランスを推定
- バランス変更履歴を ADR 化(なぜその変更を入れたか、テストで何が改善したか)

## プレイテスト

- ターゲットプレイヤー(コア層 / カジュアル層 / 子ども等)を分けてテストし、別々にログを取る
- プレイテストログは `docs/playtest-logs/<日付>-<人数>-<バージョン>.md` 形式
  - 参加者(匿名化、ADR 0001 準拠)、所要時間、勝者、各プレイヤーの感想、観察した問題
- 同じ問題が **2 回**観察されたら設計に手を入れる(1 回は偶然の可能性)
- プレイヤー間の経験差で結果が変わる(熟練者だけ強い等)はバランス問題の兆候

## ルールブック執筆

- 全体構成: コンセプト → コンポーネント → セットアップ → ターン構造 → 詳細ルール → 終了 / 勝利 → 例示 → FAQ
- 例示は**実プレイの再現**で書く(「もしA がカードXを使ったら...」)、抽象記述だけにしない
- 用語集を必ず作る(同じ概念に複数表記が混じらないよう、最初に定義)
- 矛盾チェック: 同じ事象が複数ルールで説明されていないか、優先順位が明確か
- レイアウトは複数言語化を想定(改行 / 文字数 / 図版位置)

## デジタル化を視野に入れる場合

- ルールを**プログラム可能な形**(状態遷移 / 公開情報 / 非公開情報の境界)で書く
- ランダム要素は seed 管理可能に
- アクションの可逆性(undo) / リプレイ性を最初から設計
- AI 対戦相手の難易度設計(熟練度別 thinking depth、ヒューリスティクス)

## 推奨 skill / subagent(板ゲー設計で頻出)

- ルールブック / プレイテストログ執筆: `japanese-tech-writing` skill
- バランス変更時: `adr-writing` skill(変更理由を ADR 化)
- ルールブックの矛盾検出: `code-reviewer` 相当の subagent 起動も検討余地(将来 `rulebook-reviewer` 追加候補)
