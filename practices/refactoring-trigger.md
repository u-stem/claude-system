# 共通化・抽象化の判断タイミング

## 関連する原則

- [`principles/03-skill-composition.md`](../principles/03-skill-composition.md) — 1 単位 1 責務、3 回出現で抽象化検討
- [`principles/02-decision-recording.md`](../principles/02-decision-recording.md) — 共通化の判断も記録する
- [`principles/06-evolution-strategy.md`](../principles/06-evolution-strategy.md) — テスト緑前提で構造を変える

## いつ使うか(トリガー)

- 同じパターンが 3 回現れた
- 能力単位の責務が 1 文で説明できなくなった
- プロジェクト固有の事情が共通層に染み出している
- 1 つの関数に引数のフラグで挙動を切り替える分岐が増えた
- 同じテストパターンが繰り返し書かれている
- 「ここを直すとあそこも直さないといけない」と感じる箇所が見える

## 手順

1. **テストが緑であることを確認する**: 振る舞いを保つ作業の前提。失敗しているテストがあるなら先に直す
2. **出現箇所を数える**: 3 回未満なら今は抽象化しない。早すぎる抽象化は誤った形で固定化される
3. **共通する振る舞いを言語化する**: 「3 箇所で同じ動作をしている」を 1 文で書く。書けないなら共通点が見えていないので抽象化しない
4. **抽象化の単位を決める**: その共通振る舞いを 1 つの能力単位として抽出する。引数で挙動分岐を作らない
5. **置換は 1 箇所ずつ**: 既存箇所を新しい能力単位の呼び出しに 1 つずつ置き換え、各置換でテストが緑のままを確認する
6. **振る舞いを変えない**: リファクタリング工程で機能追加・修正を同時にしない。振る舞い変更は別コミットで切る
7. **共通化を解除する道も残す**: 後で共通化が誤りだと判明したときに戻せるよう、構造変更を可逆な形で行う

## 判断基準

| 状況 | 共通化する | 待つ |
|---|---|---|
| 同じパターンが 3 回以上出現 | ◯ | |
| 共通する振る舞いを 1 文で書ける | ◯ | |
| 1〜2 回しか出現していない | | ◯ |
| 共通点がうまく言語化できない | | ◯(まだ抽象が見えていない) |
| 各箇所で微妙に挙動が違い、引数で分岐が必要 | | ◯(別々のままが良いかもしれない) |

「DRY しないと気持ち悪い」だけで動かない。出現回数と振る舞いの一致が判断軸。

## アンチパターン

- 1 回しか出ていないパターンを「将来増えるはず」で先回り抽象化する
- 振る舞い変更とリファクタリングを同じコミットに混ぜる
- テスト未整備の状態で構造を変える(振る舞い保持を確認できない)
- 引数のフラグで複数の挙動を切り替える 1 つの能力単位を作る(責務が複数化する)
- 共通化したい誘惑だけで、共通化後の置換を完了させずに途中で止める(中途半端な抽象化が残る)
- 共通化の判断根拠を残さず、後から「なぜこの構造?」が読めなくなる

## 旧資産からの継承

旧資産では「テスト緑前提のリファクタ」「コピペ実装の禁止」「振る舞いは保つ」が個別運用として書かれていた。本 practice はそれを共通化判断の起点と手順としてまとめ、原則層の能力合成・意思決定保存に紐づけたもの。
