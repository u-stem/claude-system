# 依存関係の衛生管理

## 関連する原則

- [`principles/06-evolution-strategy.md`](../principles/06-evolution-strategy.md) — 機械的ガードレールを置く、変動の速い層で自動防御する
- [`principles/02-decision-recording.md`](../principles/02-decision-recording.md) — 意図しないロックファイル変更は意思決定の不在、調査して決着をつける

## いつ使うか(トリガー)

- 新しい依存パッケージ(ライブラリ、補助ツール、外部サーバー)を追加するとき
- 既存依存のバージョンを更新するとき
- ロックファイルが意図せず変更されたとき
- 補助ツール(言語実行環境、CLI、外部サーバー)のバージョンが揺れているとき

## 手順

1. **バージョンを固定して追加する**: パッケージ追加コマンドはバージョン指定形式を使う。指定なしでは追加しない
2. **公開からの経過日数を確認する**: 公開直後のパッケージはタイポスクワッティングや乗っ取りの疑いが残る。一定日数(目安として公開後 7 日以上)を経たものを選ぶ
3. **採用前にメタ情報を確認する**: 週間ダウンロード数、メンテナンス活動、既知の脆弱性報告を一通り見る。極端に少ないダウンロード・長期未メンテ・未解決の重大報告があるものは避ける
4. **タイポを確認する**: パッケージ名のスペルを公式リポジトリで照合する。類似名による誤取得を防ぐ
5. **本番文脈での未指定実行を避ける**: 本番・CI・配布対象の実行環境では、バージョン未指定のオンザフライ実行ツールを使わない
6. **ロックファイルを意図的に保つ**: 意図しないロックファイル変更が出たらコミット前に調査する。差分の理由が説明できないままコミットしない
7. **再現可能インストールを使う**: CI や再現環境ではロックファイル尊重モード(ロック内容を逸脱した解決を禁止する形式)で取得する
8. **更新は意図的に行う**: 「最新に上げたら通った」を回避し、更新タイミングを定期化する。各更新の理由・影響範囲を変更履歴に書く

## 判断基準

| 状況 | 採用 | 採用しない / 待つ |
|---|---|---|
| 公開後 7 日未満 | | 待つ(ガードレールでブロック) |
| 週間ダウンロード数が極端に少ない | | 慎重に検討 |
| 長期未メンテ | | 代替を探す |
| バージョン指定可能 | 固定で追加 | |
| 本番文脈で未指定実行 | | 禁止 |
| ロックファイル差分の理由が不明 | | コミット前に調査 |

## アンチパターン

- バージョン未指定で追加し、後で別バージョンが入って動作が変わる
- 公開直後のパッケージを警戒なしに採用する
- パッケージ名のタイポを確認せず、類似名の悪意あるパッケージを取得する
- ロックファイルの予期しない差分を「とりあえずコミット」する
- 本番文脈で未指定バージョンのオンザフライ実行ツールを呼び出す
- 補助ツールの更新を「動かなくなったときだけ」行い、累積した変更を一度に飲み込む
- 機械的ブロック(経過日数チェック等)を「面倒だから」プロジェクト単位で恒久無効化する

## 旧資産からの継承

旧資産のセキュリティ章「依存関係(バージョン固定、タイポ対策、ダウンロード数確認、ロックファイル監視、本番での未指定実行禁止)」と、補助ツールのバージョン固定運用、公開後経過日数による自動ブロックの設計を抽象化したもの。具体ガードレールの実装は適応層・自動化層に置く。
