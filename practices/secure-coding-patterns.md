# セキュアコーディングのパターン

## 関連する原則

- [`principles/05-separation-of-concerns.md`](../principles/05-separation-of-concerns.md) — 境界で検証する、明示的許可を選ぶ
- [`principles/06-evolution-strategy.md`](../principles/06-evolution-strategy.md) — 安易な抑制を許さず根本原因を修正する

## いつ使うか(トリガー)

- 外部入力(ユーザー入力、ネットワーク経由データ、ファイル読み込み)を扱うコードを書くとき
- データベース・テンプレート・シェル・ファイルパス・ネットワーク・外部 API への出力を伴うコードを書くとき
- セキュリティ機能(検証、エンコード、エスケープ)を一時的に無効化したい誘惑が生じたとき
- レビューでセキュリティ観点の判断軸が必要になったとき

## 手順

1. **境界を識別する**: どこから先が信頼できない外部入力か、どこから先が内部信頼領域かを線引きする
2. **境界で検証する**: 形式・型・範囲・許可リストへの所属を境界 1 箇所で検査する。内部関数で都度検証しない
3. **出力先に応じてエンコードする**: 出力先のコンテキスト(構造化クエリ・テンプレート・シェル・ファイルパス・ヘッダ・ネットワーク)に応じて適切なエスケープ・パラメータ化を選ぶ
4. **構造化クエリでは値とコードを分離する**: 文字列結合でクエリを組み立てない。値は引数として渡す機構を使う
5. **テンプレート出力ではフレームワークの組み込みエスケープを使う**: 自前でエスケープを書かない
6. **ファイルパスは正規化してから範囲確認する**: 入力パスを正規化(現実のパスに解決)してから、期待するディレクトリ配下に収まることを確認する
7. **シェル実行は配列で渡す**: シェル経由でなく、引数を配列で直接渡す形式を使う。値の解釈をシェルに委ねない
8. **拒否でなく許可で防ぐ**: 「これを禁止する」の列挙でなく「これを許す」の列挙で境界を作る。列挙漏れがそのまま脆弱性になる構造を避ける

## 判断基準

| 場面 | 望ましい | 避ける |
|---|---|---|
| 構造化クエリ | パラメータ化 | 文字列結合 |
| テンプレート出力 | 組み込みエスケープ | 自前エスケープ |
| ファイルパス | 正規化 + 範囲確認 | 入力をそのまま結合 |
| シェル実行 | 配列で引数渡し | シェル経由の実行 |
| 入力受付 | 許可リスト | 拒否リスト |
| 検証位置 | 境界 1 箇所 | 内部関数で都度 |

## アンチパターン

- 構造化クエリを文字列結合で組み立てる
- テンプレート出力で自前のエスケープ関数を書く(漏れが必ず出る)
- ファイルパスを検証なしに連結し、外部入力で意図しない場所に書く
- シェルを経由した実行で、値の解釈をシェルに任せる
- 拒否リスト方式で「危険そうなパターンを除外」する(列挙漏れが脆弱性になる)
- 境界を曖昧にし、内部関数のあちこちで重複検証する(責務が散らばる)
- 検証機構を「テスト中だけ」「一時的に」無効化したまま放置する

## 旧資産からの継承

旧資産のセキュリティ章「入力バリデーション」「SQL/XSS/パストラバーサル/コマンドインジェクション対策」「拒否リストより許可リスト」「セキュリティ機能の一時的無効化禁止」を、特定言語の関数名・構文に依存しない抽象パターンとして整理したもの。特定言語固有の関数や構文ガイドは適応層の能力単位として別途整備する。
