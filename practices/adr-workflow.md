# 意思決定記録(ADR)の運用

## 関連する原則

- [`principles/02-decision-recording.md`](../principles/02-decision-recording.md) — 判断の根拠を未来の自分・未来の AI が再構成できる形で残す
- [`principles/06-evolution-strategy.md`](../principles/06-evolution-strategy.md) — 不変層への手入れは MAJOR バージョンアップ相当として記録する

## いつ使うか(トリガー)

- 不変層の原則を追加・改訂・撤回するとき
- 機械的ガードレール(機械検出ルール、自動拒否、自動承認)を新設・撤去するとき
- セキュリティ・プライバシー方針を変更するとき
- 既存ファイル群やコミット履歴の破壊的書き換えを伴う方針転換を行うとき
- 「なぜこうしなかったのか」を将来の自分が問うときに、記憶が答えにならないと予想されるとき

逆に、単純なバグ修正、typo 直し、新しい能力単位の追加、軽微な文言調整では ADR を書かない。

## 手順

1. **連番を採番する**: 直近の番号 + 1。欠番禁止、撤回しても番号は残す
2. **Status を初期値で書く**: `Proposed` か `Accepted`。即決なら `Accepted` でもよいが、議論を残したい場合は `Proposed` から始める
3. **必須セクションを埋める**: Context(なぜ必要か) → Decision(何を決めたか) → Consequences(Positive/Negative/Neutral) → Related(関連 ADR・コミット・ファイル)
4. **意思決定者を記す**: ADR 0001 の識別子規約に従う。本名・新規連絡先は書かない
5. **既存 ADR との整合をとる**: 前の決定を覆す場合、旧 ADR の Status を `Superseded by NNNN` に更新し、本 ADR 末尾の Related で参照する
6. **`meta/decisions/README.md` の一覧表に追記する**: 番号・タイトル・Status・短い概要を 1 行で

## 判断基準

| 状況 | ADR を書く | 書かない |
|---|---|---|
| 不変層の原則改訂 | ◯ | |
| 機械的ガードレールの新設・撤去 | ◯ | |
| セキュリティ・プライバシー方針の変更 | ◯ | |
| 破壊的書き換え | ◯ | |
| 単純なバグ修正 | | ◯(コミットメッセージで十分) |
| 新しい能力単位や補助エージェントの追加 | | ◯(変更履歴で十分) |
| typo・文言調整 | | ◯ |

迷ったら「3 年後の自分に聞かれて答えられるか」で決める。答えに窮しそうなら書く。

## アンチパターン

- ADR を書かずに不変層を書き換える(根拠が消え、後で誰も理由を説明できなくなる)
- Status を更新せず古い ADR を放置する(現状を反映していない判断記録は誤情報源になる)
- 番号を飛ばして採番する(欠番禁止、撤回した場合も番号は残し Status を `Withdrawn` 等にする)
- Decision だけ書き Context を省略する(なぜそうしたのかが残らない)
- Related を書かず孤立した ADR にする(関連判断との依存が読めなくなる)
- 個人特定情報を Decider 欄や本文に書き込む(ADR 0001 違反)
- Public な意思決定記録から Private リソースへの URL を貼る(ADR 0002 違反)

## 旧資産からの継承

旧資産には独立した意思決定記録の運用は存在せず、変更履歴と運用方針が同居していた。本 practice は ADR 0001/0002 の運用で得られた知見を抽象化し、原則層からの要請(意思決定の保存)を実務手順に翻訳したもの。
