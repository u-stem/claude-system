# コミット規約

## 関連する原則

- [`principles/02-decision-recording.md`](../principles/02-decision-recording.md) — 判断は事後検証可能な形で残す、コミットメッセージは Why
- [`principles/03-skill-composition.md`](../principles/03-skill-composition.md) — 1 単位 1 責務(1 コミット 1 判断単位)

## いつ使うか(トリガー)

- コミットを切るとき
- ブランチをマージするとき
- 変更履歴を更新するとき
- AI と協働した成果物を記録に残すとき

## 手順

1. **1 コミット 1 判断単位に切る**: 複数の判断を 1 つのコミットに混ぜない。混ぜると後から判断境界が読めない
2. **type を選ぶ**: 変更の性質に応じて分類する(下表)。複数 type が混ざる変更は分割を検討する
3. **件名は短く具体に**: 「何を変えたか」を一行で表す。スコープを括弧書きで補う場合は変更領域を簡潔に
4. **本文に Why を書く**: なぜこの変更が必要か、どの代替案を捨てたか、どのような影響があるかを残す
5. **認証情報の混入を確認する**: ステージング前に差分を見て、機密値・本物のキー・個人特定情報が含まれていないかを点検する
6. **指定範囲外のファイルが混じっていないか確認する**: タスクのスコープ外の変更が紛れ込んでいないかを確認する
7. **検証の証跡を残す**: 大きな変更には検証コマンドの実行結果をコミットメッセージ本文か関連ドキュメントに残す

## 判断基準

### type の分類

| type | 用途 |
|---|---|
| feat | 新機能、新しい原則・実践パターン・能力単位・補助エージェントの追加 |
| fix | バグ修正 |
| docs | ドキュメントのみの変更 |
| refactor | 振る舞いを変えない構造変更 |
| chore | 構築、自動化、雑務 |
| test | テストの追加・修正 |

### 切り分けの目安

| 状況 | 1 コミット | 分割 |
|---|---|---|
| 1 つの判断単位に収まる | ◯ | |
| feat と refactor が同居 | | ◯ |
| 機能追加と関連ドキュメント更新 | ◯(同じ判断単位) | |
| 異なる関心の修正が複数 | | ◯ |

## アンチパターン

- 1 コミットに複数の判断単位を詰め込む(後から判断境界が読み取れない)
- 件名に What だけ、本文を空欄にする(なぜ変えたかが消える)
- 認証情報や個人特定情報を確認せずコミットする(履歴に永続化する)
- スコープ外の「ついで」変更を含めて、本来の判断を埋もれさせる
- 検査やテストを通さずに区切る(検証の証跡が残らない)
- 同じ趣旨の小さな修正を細切れにコミットする(逆方向の過剰分割)
- 件名に過剰装飾(絵文字、誇張表現)を入れる
- 個人特定情報をコミット作者欄に新たに混入させる(意思決定記録の規約に違反する)

## 旧資産からの継承

旧資産の Git 章(Conventional Commits、type 分類、認証情報の混入禁止、Why ベースのメッセージ)、ドキュメント更新の同時性、コミット粒度に関する運用知見を抽象化したもの。具体的なコミットメッセージ生成補助や履歴整形のメカニズムは適応層に置く。
