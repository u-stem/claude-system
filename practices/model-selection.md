# モデル選択ガイドライン

## 関連する原則

- [`principles/01-context-economy.md`](../principles/01-context-economy.md) — 注意総量は有限、タスクに応じて適切な投資をする
- [`principles/06-evolution-strategy.md`](../principles/06-evolution-strategy.md) — モデル世代は変動する、固定的な銘柄ルールを書かず複雑度で判断する

## いつ使うか(トリガー)

- 新しい能力単位や補助エージェントを設計するとき
- 既存能力単位の推奨モデルを見直すとき
- コスト・速度・品質のトレードオフが顕在化したとき
- モデル世代の更新で従来の選択基準が陳腐化したとき

## 手順

1. **タスクの複雑度を判定する**: 単純な実行(取得・整形・列挙)か、判断を要する思考(設計・レビュー・原子性・セキュリティ)か
2. **思考主体か実行主体かを切り分ける**: 思考主体なら上位モデル、実行主体なら軽量モデル
3. **コスト感度を確認する**: 同じ作業を高頻度で回すなら、軽量側を優先。重要度が高く頻度が低いなら上位側を許容
4. **能力単位に推奨モデルレベルを明示する**: 「軽量」「中位」「上位」のような相対水準で記述する。固有銘柄名は適応層に閉じ込める
5. **逸脱を許容する**: 推奨は推奨に留め、利用者がコスト・速度の事情で変更できる余地を残す
6. **逸脱の根拠を残す**: 標準と異なる選択をした能力単位は、その理由を能力単位の説明に書く
7. **モデル更新時に見直す**: 新世代が出たときに既存の推奨水準が妥当か再評価する

## 判断基準

| タスクの性質 | 推奨水準 |
|---|---|
| データ取得・結果整形・既存出力の要約 | 軽量 |
| 単純なロジック実装、雛形コード生成 | 軽量〜中位 |
| 一般的なリファクタリング、テスト作成 | 中位 |
| アーキテクチャ設計、セキュリティレビュー、原子性が問われる処理 | 上位 |
| コードレビュー、ドキュメントの整合判定 | 中位〜上位 |
| 探索だけが重く判断が薄い場合 | 軽量(コンテキスト消費を抑える側面で) |

固定的に「レビューは上位 / 実装は中位」のような銘柄ルールを書かない。あくまで複雑度と思考量で判断する。

## アンチパターン

- 役割で固定ルール化する(「レビューは常に上位、実装は常に中位」のような硬直)
- 全タスクを最高位モデルで回す(コストが過剰、速度も犠牲になる)
- 全タスクを軽量モデルで回す(複雑判断で品質が崩れる)
- 推奨モデルを固有銘柄名で不変層に書き込む(モデル世代の更新で時代遅れになる)
- 逸脱の根拠を残さず能力単位ごとにバラバラの選択をする(後から方針が読めない)
- モデル世代の更新を機に既存能力単位の推奨水準を見直さない(古い基準が固定化する)

## 旧資産からの継承

旧資産では「特定タスクは上位モデル、それ以外は中位」のような柔軟な指針が書かれていた一方、特定モデル世代の銘柄名や挙動も同居していた。本 practice は前者(複雑度ベースの判断軸)を抽象側に切り出し、後者(銘柄・挙動)を適応層に閉じ込める方針として整理したもの。
