# 開発ワークフロー

## 関連する原則

- [`principles/02-decision-recording.md`](../principles/02-decision-recording.md) — 検証なしで完了と言わない、AI 出力は検証する、未決を未決のまま残さない
- [`principles/06-evolution-strategy.md`](../principles/06-evolution-strategy.md) — 機械的検査を組み込む、抑制の安易な選択を許さない

## いつ使うか(トリガー)

- 機能追加・バグ修正・改修に着手するとき
- AI と協働してコードを書くとき
- 複数の小さな変更を連続して入れるとき
- 完了の判断を下す直前

## 手順

1. **仕様を確定させる**: 何ができれば完了か、何が完了の証拠か、どこまでがスコープかを最初に明確にする
2. **既存パターンを確認する**: 似た処理が既にあるなら踏襲する。新規にパターンを増やさない
3. **テストから書く**: テスト戦略の practice に従って、まず失敗するテストを書く
4. **小さく実装する**: テストを通す最小限の実装にとどめる。先回りで関連機能を盛り込まない
5. **静的検査をかける**: 整形・型検査・静的解析を実装の合間に通す。最後にまとめてではなく、変更の小さな粒度で都度
6. **検証して初めて完了と言う**: 静的検査・テスト・必要なら手動確認の各証跡が揃ってから完了とする。「動くはず」「たぶん通る」では完了にしない
7. **小さくコミットする**: 1 判断単位で 1 コミット。変更の境界が読める粒度で記録を残す
8. **関連ドキュメントを同じコミットで更新する**: 仕様・README・能力単位の説明など、コードと整合する文書を一緒に直す
9. **未解決を残さない**: 解消すべき検査エラーや未確認事項を残したままセッションを閉じない。残す場合は引き継ぎとして構造化記録に残す

## 判断基準

| 場面 | 進める | 止まる |
|---|---|---|
| 仕様が言語化できている | ◯ | |
| 完了基準が明確 | ◯ | |
| 検証の証跡が揃っている | ◯ | |
| 仕様が曖昧 | | ◯(先に明確化) |
| 「たぶん通る」と感じる | | ◯(検証してから判断) |
| 複数の変更が混ざりそう | | ◯(切り分ける) |

## アンチパターン

- 仕様未確定のまま実装を進め、完了基準が後付けになる
- 大量の変更を 1 コミットに詰め込み、判断単位が読み取れなくなる
- 検証を最後にまとめて行い、途中で壊れた箇所が特定できなくなる
- 「たぶん大丈夫」を確認も明言もせず完了宣言する
- 静的検査の警告を「一時的に」無効化したまま放置する
- 実装と関連ドキュメントの更新を別コミットに分け、片方を忘れる
- AI 生成コードを検証なしで採用する
- 完了基準を満たさないまま「とりあえず動く」を理由に区切る

## 旧資産からの継承

旧資産のワークフロー(仕様確認 → TDD → 型検査・静的解析 → 小さくコミット)、禁止事項(TODO 先送り、たぶん大丈夫、コピペ実装)、ドキュメント更新の同時性、検証主義(AI 出力を検証する)を統合したもの。具体的な検査コマンドや設定は適応層に閉じ込める。
