# テスト戦略

## 関連する原則

- [`principles/03-skill-composition.md`](../principles/03-skill-composition.md) — テストは実装でなく振る舞いを記述する
- [`principles/02-decision-recording.md`](../principles/02-decision-recording.md) — テストは振る舞いの仕様を保存する記録、AI 出力の検証手段でもある
- [`principles/05-separation-of-concerns.md`](../principles/05-separation-of-concerns.md) — テストの境界(unit/integration/e2e)は関心の粒度で分ける

## いつ使うか(トリガー)

- 新機能を実装するとき(テストから書く)
- バグを修正するとき(再現テストから書く)
- リファクタリングを行うとき(緑のテストを前提に構造を変える)
- AI が生成したコードを採用するとき(検証の手段)

## 手順

### TDD サイクル

1. **Red**: まず失敗するテストを書く。失敗の出力が想定通りであることを確認
2. **Green**: テストが通る最小限の実装を書く。それ以上は書かない
3. **Refactor**: テスト緑のまま構造を整える。振る舞いは変えない

### テスト設計

1. **振る舞いベースで命名する**: 「対象が条件下でどう動くか」を記述する。実装関数名で命名しない
2. **1 テスト 1 アサーション**: 1 つのテストで 1 つの振る舞いだけ確認する
3. **Arrange-Act-Assert**: 準備 → 実行 → 検証を 1 テスト内で順序立てる
4. **テスト内に分岐・繰り返しを書かない**: 条件分岐や反復はテスト対象側に置き、テスト本体は直線的にする
5. **境界を選ぶ**: 純粋なロジックは単体テスト、外部 I/O との結合は統合テスト、利用者向けフローは終端テスト
6. **データは現実的かつ最小**: テスト用データはテストの意図を表すのに必要な最小限にとどめる
7. **可変な共有状態を持たない**: テスト間で副作用が伝播する状態は作らない
8. **境界ではモックしない**: 統合テストで実物の DB・API を使う。境界の挙動こそテストの対象

## 判断基準

| 対象 | テスト対象 | テスト不要 |
|---|---|---|
| 純粋ロジック | ◯(単体) | |
| 外部 I/O との結合 | ◯(統合、境界では実物) | |
| 利用者フロー | ◯(終端、最小数) | |
| フレームワーク内部 | | ◯ |
| 自明な値の保管・取得 | | ◯ |
| サードパーティライブラリの挙動 | | ◯ |

## アンチパターン

- 失敗するテストを書かずに実装から始める(Red を踏まないと Green の意味が薄れる)
- 1 テストで複数の振る舞いをまとめて検証する(壊れたときに原因が特定できない)
- 実装の関数名でテストを命名する(リファクタで一斉に壊れる)
- スキップ・無視・除外で失敗テストを残す(直すか削除する)
- 常に通るトートロジーテスト(実装に関わらず pass する)
- 過剰なモックで対象コードを実質テストしていない状態(モック同士の整合だけ確認している)
- 統合テストで境界をモックする(境界の挙動が検証されない)
- テストを書かずに AI 生成コードを採用する(検証行為が抜ける)

## 旧資産からの継承

旧資産の TDD サイクル(Red-Green-Refactor)、テスト命名(振る舞い記述)、1 アサーション 1 テスト、Arrange-Act-Assert、境界(unit/integration/e2e)の使い分け、テストデータ規約、禁止パターン(スキップ・トートロジー・過剰モック)を統合し、原則層への参照を明示化したもの。
