# 新規プロジェクト立ち上げ

## 関連する原則

- [`principles/05-separation-of-concerns.md`](../principles/05-separation-of-concerns.md) — 粒度を混ぜない、外側の都合を内側に持ち込まない
- [`principles/06-evolution-strategy.md`](../principles/06-evolution-strategy.md) — 機械的ガードレールを最初から置く、不変層への依存を一方向に保つ

## いつ使うか(トリガー)

- 新規プロジェクトを開始するとき
- 既存プロジェクトを本システムの管理下に取り込むとき
- 既存プロジェクトの設定を本システムの構造に合わせ直すとき

## 手順

1. **層の役割を確認する**: 不変層 = 普遍的原則、抽象層 = 実践パターン、適応層 = ツール固有、個別層 = プロジェクト固有
2. **共通基盤を参照する形で始める**: 不変層・抽象層・適応層は新プロジェクトにコピーしない。参照(取り込み)で済ませる
3. **個別事情を個別層に隔離する**: プロジェクト固有の値(サービス名、内部ドメイン名、固有のディレクトリ規約)はプロジェクトリポジトリ内に閉じ込める
4. **配布断片を組み込む**: 共通基盤側が用意する断片(雛形、テンプレート、設定の部分集合)を取り込み、個別箇所だけ上書きする
5. **機械的ガードレールを最初に置く**: 機密情報の検出、危険な操作の拒否、依存関係の経過日数チェックといった自動防御を初日に組む
6. **意思決定の保存先を用意する**: プロジェクト固有の意思決定記録ディレクトリを最初に作る。後付けで作るとそれまでの判断が記録されない
7. **検証コマンドを 1 行で書ける状態にする**: 静的検査・テスト・整形をひとまとめで実行できる入口を 1 つ用意する
8. **取り込み撤回手順を確認する**: 万一、共通基盤との接続を解除したくなった場合の戻し方を最初に確認しておく(冪等性の確認)

## 判断基準

| 配置先 | 何を置くか |
|---|---|
| 不変層(共通基盤) | 普遍的原則(ツールが変わっても残るもの) |
| 抽象層(共通基盤) | 実践パターン(複数プロジェクトに通用するもの) |
| 適応層(共通基盤) | 特定ツール向けの設定・拡張・能力単位 |
| 個別層(プロジェクト) | プロジェクト固有の値、固有の設定上書き、固有の補助スクリプト |

迷ったら「他プロジェクトでも使うか?」で判断する。1 つでも使うなら共通側、自プロジェクトでしか使わないなら個別側。

## アンチパターン

- 共通基盤を直接コピーして個別カスタマイズを加える(変更が伝播しなくなる)
- 個別事情を共通基盤に書き戻す(他プロジェクトに不要な分岐が増える)
- 機械的ガードレールを後回しにする(その間に事故が起きる)
- 意思決定記録の置き場を後から作る(立ち上げ期の判断が散逸する)
- 検証コマンドの入口を統一せず、能力単位ごとに別コマンドにする(忘れられて検査されなくなる)
- 取り込み撤回の手順を考えずに依存を増やす(冪等性が崩れる)

## 旧資産からの継承

旧資産には新規プロジェクト立ち上げの抽象手順は明文化されておらず、個別 setup スクリプトに散在していた。本 practice はそれを抽象側に切り出し、層構造の運用ルールとして整理したもの。
