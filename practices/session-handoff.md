# セッション跨ぎでの引き継ぎ

## 関連する原則

- [`principles/01-context-economy.md`](../principles/01-context-economy.md) — 注意は有限資源、タスク間でコンテキストを区切る
- [`principles/02-decision-recording.md`](../principles/02-decision-recording.md) — 判断と未決を保存しないと再構成不能になる

## いつ使うか(トリガー)

- 1 セッションで完結しないタスクの途中で打ち切るとき
- 別セッション(別人、または時間経過後の自分)に作業を渡すとき
- 大きな調査の結論だけを次セッションに渡したいとき
- セッション末で自動的に古いコンテキストを失う前

## 手順

1. **完了済みを書き起こす**: 何が確定したか、どのファイルがどう変わったか。コミット ID があるなら明示
2. **未完を書き起こす**: 何が残っているか、どこで止まったか、止まった理由(検討中・未確認・要承認 等)
3. **前提条件を残す**: 環境変数、参照する設定ファイル、想定するブランチ・ディレクトリ
4. **検証コマンドを記録する**: 受け手が「現状が想定通りか」を即座に確認できるコマンド列
5. **次の最初の行動を指示する**: 「最初に何を読み、最初に何を実行するか」を 1〜3 ステップで具体化
6. **構造化して残す**: 自然言語の長文に埋めず、設定ファイル相当の構造化された形(キー・値、または見出し付きセクション)で記録する
7. **置き場所を固定する**: プロジェクト固有の決まった場所に置き、受け手が探さなくて済む
8. **完了したら破棄する**: 引き継ぎ用ファイルは作業完了後に削除する。古いものを残すと誤参照を招く

## 判断基準

| 状況 | 引き継ぎを書く | 書かない |
|---|---|---|
| タスクが 1 セッションを超える | ◯ | |
| 別セッションが本格的に作業する | ◯ | |
| 結論を再現可能な形で伝えたい | ◯ | |
| 数分の中断で同一セッションに戻る | | ◯(オーバーヘッドが価値を超える) |
| 単発の小さい修正で完結 | | ◯ |

## アンチパターン

- 未完だけ書き、完了済みを書かない(受け手が同じ調査を繰り返す)
- 自然言語の長文を書き、構造化せず読みにくい(信号比が下がる)
- 検証コマンドを書かず「動くはず」とだけ書く(受け手が現状確認できない)
- 引き継ぎファイルを作業完了後も残す(古い情報が次タスクの誤参照を招く)
- 「次にすること」を抽象的に書き、具体的な最初の行動を示さない
- 個人特定情報を引き継ぎファイルに書く(ADR 0001 違反)

## 旧資産からの継承

旧資産では引き継ぎ用設定ファイルを置く運用があった(構造化された形で完了済み・未完・前提・検証手順を書く方式)。本 practice はその運用を、ツール固有の API に依存しない手順として抽象化したもの。
