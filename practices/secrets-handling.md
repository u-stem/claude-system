# 認証情報・秘密鍵の取り扱い

## 関連する原則

- [`principles/05-separation-of-concerns.md`](../principles/05-separation-of-concerns.md) — 設定値と機密情報は別の場所に分離する
- [`principles/06-evolution-strategy.md`](../principles/06-evolution-strategy.md) — 一時的な無効化を許さず、機械的ガードレールで防ぐ

## いつ使うか(トリガー)

- 認証情報・API キー・秘密鍵・パスワードを扱うコードや設定を書くとき
- 新しい外部サービスと連携するとき
- 機密情報を誤って履歴に入れた可能性を発見したとき
- 機密情報の保管場所を選定するとき

## 手順

1. **保管場所を分離する**: 設定ファイルに直書きしない。環境変数、専用シークレット管理機構、または OS のキーチェーンを使う
2. **境界を明示する**: 機密が存在するファイル・ディレクトリと、そうでないものを構造的に分ける。両者を同じ場所に置かない
3. **ログ・エラー出力に出さない**: 例外メッセージ、デバッグ出力、解析用ダンプに値を含めない。値を識別子(マスクした参照子)で表す
4. **テストにも本物を使わない**: テスト用ダミー値を用意し、本番のキーをテストファイルに書かない
5. **機械的ブロックを設定する**: 機密相当のファイル名・拡張子(環境設定ファイル、各種鍵ファイル等)への書き込みを自動拒否する仕組みを入れる
6. **コミット履歴を機械検出する**: 機密が混入していないかを履歴全体に対して機械的に走査する仕組み(コミット前 hook、CI)を組む
7. **露出時は即ローテーションする**: ローカルコミットでも、リモートに出ていなくても、履歴に残った時点でローテーションする
8. **無効化を禁ずる**: 機械的ガードレールを「一時的に」無効化したまま運用しない。無効化が必要な状況は、設計が間違っているサインと捉える

## 判断基準

| 配置先 | 機密情報を置く | 置かない |
|---|---|---|
| 設定ファイル(配布対象) | | ◯ |
| 環境変数 | ◯ | |
| 専用シークレット管理機構 | ◯ | |
| OS キーチェーン | ◯ | |
| ソースコード | | ◯ |
| テストコード | | ◯(ダミー値で代替) |
| ログ出力 | | ◯ |
| バージョン管理履歴 | | ◯ |

## アンチパターン

- 設定ファイルに本物のキーを書き込み、配布対象に含める
- ログ出力に認証情報の値をそのまま出す(後でログを共有したときに漏れる)
- テストコードに本番のキーを書き、テスト履歴に残す
- 露出を確認しても「リモートに出ていないから大丈夫」とローテーションを省略する
- 機械的ブロックを「面倒だから」恒久的に無効化する
- 機密が含まれた可能性のあるコミットを「reset で消したから安全」と思い込む(履歴は他者の手元に残り得る)
- 機密と通常の設定値を同じディレクトリに混在させ、配布範囲の判定を曖昧にする

## 旧資産からの継承

旧資産のセキュリティ章「シークレット(環境変数で扱う、ログに出さない、露出時はローテーション、一時的無効化禁止)」を、抽象境界の運用と機械的ガードレールの導入として整理したもの。具体的な機密検出ルールやファイルパターンは適応層・自動化層に置く。
