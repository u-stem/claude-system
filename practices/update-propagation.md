# 共通設定変更の波及判断

## 関連する原則

- [`principles/06-evolution-strategy.md`](../principles/06-evolution-strategy.md) — 不変層と変動層で変更コストを変える
- [`principles/05-separation-of-concerns.md`](../principles/05-separation-of-concerns.md) — 粒度ごとの伝播範囲を分ける
- [`principles/02-decision-recording.md`](../principles/02-decision-recording.md) — 大きな変更は判断を記録する

## いつ使うか(トリガー)

- 共通基盤の設定・能力単位・補助エージェント・ガードレールを変更したとき
- 既存プロジェクトに変更を反映するか判断するとき
- 変更が一部プロジェクトでのみ望ましく、全体には不要と感じたとき
- 大規模な共通基盤改訂の後、各プロジェクトの追従計画を立てるとき

## 手順

1. **影響範囲を層で識別する**: 変更が不変層 / 抽象層 / 適応層 / 個別層のどこに属するかを判定する
2. **必要な伝播範囲を決める**: 層によって伝播範囲が異なる(下表を参照)
3. **意思決定を記録する**: 不変層変更は ADR、それ以外は変更履歴に書く
4. **段階的に伝播させる**: 一度に全プロジェクトに反映しない。1 プロジェクトで検証 → 次に展開、の順
5. **個別プロジェクトの判断を尊重する**: 抽象層・適応層の変更は、各プロジェクトで採否を判断できる余地を残す
6. **取り込み撤回手順を残す**: 変更が問題を生んだ場合に戻せるようにする(冪等性)
7. **各プロジェクトの状態を可視化する**: どのプロジェクトがどのバージョンの共通基盤を取り込んでいるかを記録する

## 判断基準

| 変更が属する層 | 伝播範囲 | 理由 |
|---|---|---|
| 不変層 | 全プロジェクト(MAJOR 相当) | 普遍原則の改訂は全体に効く |
| 抽象層(実践パターン) | 各プロジェクトの判断 | 採否はプロジェクトごとの事情に依存 |
| 適応層(特定ツール向け) | 当該ツールを利用するプロジェクトのみ | 他ツール利用プロジェクトには無関係 |
| 個別層(プロジェクト固有) | そのプロジェクトのみ | 他プロジェクトに伝播してはいけない |

迷ったら「他プロジェクトに勝手に反映されたら困るか?」を逆方向から考える。困るなら個別側、困らないなら共通側。

## アンチパターン

- すべての変更を全プロジェクトに即座に強制反映する(個別事情を無視して破壊する)
- 変更を完全放置し、各プロジェクトが古い共通基盤に固定される(意図しない長期分岐)
- 不変層の変更に意思決定記録を残さない(後で経緯が辿れない)
- 取り込み撤回手順を用意しないまま依存を増やす(問題発生時に戻せない)
- 個別プロジェクトの状態(どのバージョンを取り込んでいるか)を可視化しない(全体像が見えなくなる)
- 適応層の変更を不変層と同じ重さで扱う(変更コストが過剰になり改善が止まる)

## 旧資産からの継承

旧資産には共通設定変更の伝播判断を扱う独立した章はなく、各 setup スクリプトと運用慣習に散在していた。本 practice はそれを層構造に紐づけて整理し、伝播範囲の判断を明示化したもの。
